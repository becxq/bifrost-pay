# bifrost-pay

`bifrost-pay` — это высокопроизводительный **gRPC-сервис идемпотентности**, разработанный для предотвращения дублирования критических операций (например, финансовых платежей). Сервис построен на базе двухуровневой архитектуры хранения данных (**Redis + PostgreSQL**) и обеспечивает надежную защиту от повторных запросов в распределенных системах.

---

## Требования

Перед запуском убедитесь, что в вашей системе установлены:
* **Go** версии `1.25.6`
* **Docker** и **Docker Compose**

---

## Установка и запуск

### 1. Запуск инфраструктуры
Для развертывания необходимых баз данных в бэкграунде выполните команду:
```bash
docker-compose up -d
```

Это автоматически поднимет и настроит:
* **PostgreSQL 16** — доступен на внешнем порту `:5433`
* **Redis 7** — доступен на порту `:6379`

### 2. Запуск сервиса
После успешного старта инфраструктуры запустите сам gRPC-сервис:
```bash
go run main.go
```
После этого сервис начнет принимать бинарные запросы на порту **`:50051`**.

## API Спецификация (`IdempotencyService`)

Сервис предоставляет два ключевых RPC-метода для управления жизненным циклом идемпотентных операций:

### 1. Метод `CheckKey`
Проверяет, выполнялась ли операция с указанным ключом ранее, и устанавливает распределенную блокировку для атомарного отсечения параллельно летящих дубликатов.

* **Request (`CheckKeyRequest`):**
  ```protobuf
  message CheckKeyRequest {  
    string key = 1;  
  }
  ```
* **Response (`CheckKeyResponse`):**
  ```protobuf
  message CheckKeyResponse {  
    string status = 1;  // Возможные статусы: "success", "pending", "not_found", "failed"  
    int32 code = 2;  
    string body = 3;  
  }
  ```

### 2. Метод `ConfirmKey`
Сохраняет финальный результат выполнения операции (после ответа внешнего провайдера/БД) и обновляет кэширующее хранилище L1.

* **Request (`ConfirmKeyRequest`):**
  ```protobuf
  message ConfirmKeyRequest {  
    string key = 1;  
    string status = 2;  // Принимает значения: "success" или "failed"  
    int32 code = 3;  
    string body = 4;  
  }
  ```
* **Response (`ConfirmKeyResponse`):**
  ```protobuf
  message ConfirmKeyResponse {  
    bool success = 1;  
  }
  ```

## Архитектура

### Двухуровневое хранение данных (Two-Tier Storage)

| Уровень | Технология | Назначение | Специфика реализации |
| :--- | :--- | :--- | :--- |
| **L1 Cache** | Redis 7 | Мгновенный поиск активных статусов и организация распределенных блокировок (`SetNX`). | Ключи с префиксами `cache:` и `lock:` |
| **L2 Storage** | PostgreSQL 16 | Персистентное (надежное и долговечное) хранение записей о транзакциях. | Таблица `idempotency_key` |

### Отказоустойчивость (Resilience)
В сервисе заложена отказоустойчивая логика работы в режиме хаоса: при полной недоступности L1-кэша (сбой инстанса Redis) система автоматически активирует паттерн **Fallback** и переключает обработку всех запросов напрямую на **PostgreSQL**. Это гарантирует непрерывность проведения бизнес-операций.

## Тестирование

В проекте реализованы полноценные интеграционные тесты для проверки поведения системы в многопоточной среде и при отказах инфраструктуры.

### Запуск тестов:
```bash
go test -v ./...
```

### Реализованные сценарии:
* **`TestCheckKey_Concurrency`** — Стресс-тест конкурентного доступа. Эмулирует одновременную атаку горутин с идентичными ключами и проверяет, что распределенная блокировка пропускает ровно 1 запрос, возвращая остальным 9 статус `pending`.
* **`TestCheckKey_RedisFailureFallback`** — Имитирует аварийное падение Redis в процессе выполнения бизнес-логики и проверяет корректность автоматического переключения gRPC-методов на PostgreSQL.
